Approach to, Writing the Web

  1. Web Design - Vanilla Approach

    This approach is what I call, keeping the web on a diet. This is done by using the vanilla web technologies, the base layer being a plain and simple HTML doing what it is meant to do. Content drives engagement and action

First, content engages readers and drives them to take the actions necessary to fulfill a site’s goals. This is affected by both the content itself (the writing), and how it’s presented (the typography and structural elements). Dull, lifeless, and overlong prose rarely keeps visitors’ attention for long. Short, snappy, and intriguing content grabs them and gets them to click through to other pages. Even if your pages need a lot of content — and often, they do — properly “chunking” that content by breaking it up into short paragraphs supplemented by visuals can help it keep a light, engaging feel.

Launch

Now it’s time for everyone’s favorite part of the website design process: When everything has been thoroughly tested, and you’re happy with the site, it’s time to launch. Don’t expect this to go perfectly. There may be still some elements that need fixing. Web design is a fluid and ongoing process that requires constant maintenance. Web design — and really, design in general — is all about finding the right balance between form and function. You need to use the right fonts, colors, and design motifs. But the way people navigate and experience your site is just as important.

Tools of my Craft

My approach towards web development is to make things as simple as possible. Therefore I choose to work with core web technologies that has powered the web since it infancy. Having said that I avoid using and adding unwanted layers of abstraction and complexities to my code. I write standard based semantic code which can be accessible and is easy to maintain and search-engine friendly. I often tend to breakdown my webpages into smaller pieces called components and build them in a way that can exist outside the context of the design they first appeared in. I initiate my projects by creating a style guide. So that the others involved can rely on the same to understand a project’s language and produce a more consistent user experience.
